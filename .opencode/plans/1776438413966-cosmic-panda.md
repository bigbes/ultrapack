# Plan: OpenCode Support With Amp-Derived Agents

## Goal

Generate an OpenCode-compatible `.opencode/` layer from the existing Claude Code plugin under `plugins/up/`, while preserving the Claude plugin as the source of truth. The OpenCode layer should keep skills as first-class skills, add Amp-inspired codebase-understanding agents, and update reflection guidance for portable `AGENTS.md`-first projects.

## Current Inputs

- Claude marketplace manifest: `.claude-plugin/marketplace.json`
- Claude plugin manifest: `plugins/up/.claude-plugin/plugin.json`
- Existing commands: `plugins/up/commands/{make,try,step-back,summary,reflect}.md`
- Existing agents: `plugins/up/agents/{explorer,implementer,researcher,reviewer,summarizer}.md`
- Existing skills: `plugins/up/skills/*/SKILL.md`
- Amp prompt reference: `amp_system_prompts.json`

## OpenCode Targets

OpenCode supports the same three primitives this pack needs:

- Commands: `.opencode/commands/<name>.md`
- Agents: `.opencode/agents/<name>.md`
- Skills: `.opencode/skills/<name>/SKILL.md`

Do not flatten skills into a single giant agent. OpenCode discovers `SKILL.md` files on demand, so the lowest-risk port is a near 1:1 generated layer.

## Naming

Use unprefixed OpenCode command names because command filenames become slash commands:

- `/make`
- `/try`
- `/step-back`
- `/summary`
- `/reflect`
- `/explain-diff`

Generated prompt text should keep the human-facing identity as `up` / ultrapack where useful, but internal OpenCode invocations should use OpenCode names:

- Claude source: `up:uexecute`, `/up:make`, `up:reviewer`
- OpenCode output: `uexecute`, `/make`, `@reviewer`

Avoid `up-*` command names unless namespace collisions become a real problem. OpenCode custom commands can override built-ins, so the generator should fail if a generated command name is known to collide with an OpenCode built-in.

## Architecture

### 1. Generate skills 1:1

Copy each Claude skill directory to `.opencode/skills/<name>/SKILL.md` with minimal rewrites:

- Keep `name` and `description`.
- Ensure `name` is lowercase alphanumeric with single hyphen separators and matches its directory.
- Add `compatibility: opencode` only if useful; OpenCode ignores unknown fields but recognizes `compatibility`.
- Rewrite cross-references:
  - `up:udesign` → `udesign`
  - `/up:make` → `/make`
  - `plugins/up/skills/_brevity.md` → `.opencode/skills/_brevity.md` is invalid because `_brevity` is not a valid OpenCode skill name; see reference docs below.

Reference docs with invalid skill names need special handling:

- `_brevity.md` → `.opencode/references/brevity.md`
- `_principles.md` → `.opencode/references/principles.md`

Generated skills that refer to these files should use `.opencode/references/brevity.md` and `.opencode/references/principles.md`.

### 2. Generate existing agents

Copy existing Claude agents to `.opencode/agents/` and convert frontmatter.

Common conversion:

- Remove `name:` because OpenCode uses the filename.
- Set `mode: subagent`.
- Prefer omitting `model:` by default so subagents inherit the invoking primary agent's model.
- Use `permission`, not legacy `tools`, for new OpenCode output.
- Rewrite Claude tool names in prose only when they are product-specific.

Permission mapping:

- Read-only agents (`explorer`, `researcher`, `reviewer`, `summarizer`, `finder`, `librarian`, `oracle`, `diff-explainer`):
  - `edit: deny`
  - `bash: ask` unless the agent is strictly no-shell
  - `webfetch: ask` for agents that may need external docs
- Writer agent (`implementer`):
  - `edit: ask`
  - `bash: ask`
  - `webfetch: deny` unless a phase explicitly needs it

Do not depend on OpenCode's deprecated `tools` field except as a fallback if current OpenCode rejects nested `permission` in markdown frontmatter.

### 3. Generate existing commands

Copy existing commands to `.opencode/commands/` with frontmatter conversion:

- `make.md`: `agent: build` or omitted; it should run in the current primary agent unless a dedicated `make` primary agent is added later.
- `try.md`: current agent; fastest verification path.
- `step-back.md`: current agent; no edits until user confirms the new direction.
- `summary.md`: needs OpenCode-specific rewrite; do not keep Claude JSONL transcript scraping.
- `reflect.md`: update routing to `AGENTS.md` first; details below.

Use `$ARGUMENTS` for command arguments in generated OpenCode prompts. The Claude command bodies currently refer to "the user's description follows the command"; OpenCode output should explicitly say "Use `$ARGUMENTS` as the task description."

## New Amp-Derived Pieces

### 1. `librarian` agent

Source inspiration: Amp's Librarian role in `amp_system_prompts.json` and Amp manual.

Create `.opencode/agents/librarian.md`.

Purpose:

- Deep codebase understanding across large repos or multiple repos.
- Explain feature flows end to end across repositories.
- Investigate upstream libraries/frameworks by reading source when local docs are insufficient.
- Use commit history to explain why behavior changed.
- Produce Mermaid diagrams when they materially clarify architecture.

Relationship to existing agents:

- `explorer`: local, compact, 3-5 essential files.
- `finder`: local, fast, files/ranges only.
- `researcher`: external docs/web plus synthesis.
- `librarian`: deep source archaeology, especially cross-repo or upstream source.

Constraints:

- Read-only.
- No implementation advice unless asked.
- Link every file/repo reference.
- State what repos/branches were inspected.
- If remote/private repository access is unavailable in OpenCode, say so and fall back to local repo plus public web/source links.

### 2. `finder` agent

The user wrote `findex`; treat that as `finder` unless they confirm they want a distinct name. Do not generate `findex` by default.

Create `.opencode/agents/finder.md`.

Purpose:

- Fast, parallel code search.
- Return relevant files and line ranges only.
- No architecture essay, no fix proposal, no implementation.

Use when:

- A command/agent needs likely edit locations.
- The user asks "where is X implemented?"
- `uexecute` needs a quick phase-scoping lookup before dispatching `implementer`.

Output:

```markdown
<1-2 line answer>

Relevant files:
- [path#Lx-Ly](file:///absolute/path#Lx-Ly)
```

### 3. `explain-diff` command + `diff-explainer` agent

Create:

- `.opencode/commands/explain-diff.md`
- `.opencode/agents/diff-explainer.md`

Purpose:

- Explain what changed and why it matters.
- Different from `reviewer`: it teaches, it does not judge.
- Start with behavior-level before/after.
- List "read first" files in review order.
- Walk non-trivial hunks in dependency order.
- Call out real risks only when needed for understanding.

Default diff target:

- If `$ARGUMENTS` is empty, explain current tracked/untracked checkout changes.
- If `$ARGUMENTS` is present, treat it as the diff scope or git revision phrase.

Implementation guidance:

- Prefer `git diff --merge-base origin/HEAD` for branch work when available.
- Include untracked files with `git ls-files --others --exclude-standard`.
- Avoid shell syntax in generated git command snippets where possible; OpenCode can run commands itself, but prompts should not depend on shell-only command generation.

### 4. `oracle` agent

Create `.opencode/agents/oracle.md`.

Purpose:

- Senior engineering second opinion for architecture, planning, debugging strategy, and tradeoff-heavy decisions.
- Gives one primary recommendation with at most one materially different alternative.
- Optimizes for the simplest viable path, maintainability, developer time, and risk.

Relationship to existing agents:

- `reviewer`: defect-oriented review after implementation.
- `researcher`: external knowledge synthesis.
- `librarian`: source archaeology and architecture explanation.
- `oracle`: decision support before action.

Use from:

- `udesign` for ambiguous architecture decisions.
- `uplan` for medium/large plan sanity checks.
- `uexecute` after repeated failed attempts or structural blockers.
- `/step-back` when proposing a fundamentally new direction.

Constraints:

- Read-only.
- No implementation unless explicitly asked.
- No broad option matrix.
- Include rough effort signal (`S`, `M`, `L`, `XL`) and revisit triggers.

### 4. `reflect` update

Update generated `.opencode/commands/reflect.md` semantics:

- Route project-wide guidance to `AGENTS.md` first.
- Use `CLAUDE.md` only when preserving Claude compatibility or when the repo already treats it as canonical.
- For dual support, prefer the migration pattern:
  - canonical guidance in `AGENTS.md`
  - `CLAUDE.md` as a symlink or short pointer when the filesystem/platform supports it
- Never write user/global memory unless OpenCode/Amp memory location is explicitly configured or the user approves.
- Show proposed guidance-file changes before applying them.
- Continue routing durable project facts to `README.md` or `docs/**/*.md` when they belong to project documentation rather than agent behavior.

This makes `reflect` portable across OpenCode, Amp, and Claude-style repos without hard-coding Claude as the only guidance destination.

## Generator Design

Create `scripts/generate-opencode.py`.

Requirements:

- Python 3 stdlib only.
- Idempotent: same input tree produces same `.opencode/` output.
- Do not delete `.opencode/node_modules`, `.opencode/package.json`, `.opencode/package-lock.json`, or `.opencode/plans/` if present.
- Regenerate only owned paths:
  - `.opencode/agents/`
  - `.opencode/commands/`
  - `.opencode/skills/`
  - `.opencode/references/`
- Preserve the Claude source tree untouched.

Suggested structure:

1. Read Claude commands, agents, skills, and reference docs.
2. Parse simple YAML frontmatter with a small local parser.
3. Apply target-specific rewrites.
4. Add Amp-derived generated files from templates embedded in the script or stored under `templates/opencode/`.
5. Write deterministic files with trailing newlines.
6. Print a short manifest of generated files.

## Rewrite Rules

Global generated-text rewrites:

- `/up:make` → `/make`
- `/up:try` → `/try`
- `/up:step-back` → `/step-back`
- `/up:summary` → `/summary`
- `/up:reflect` → `/reflect`
- `up:<agent-or-skill>` → `<agent-or-skill>`
- `Task tool` / `subagent_type` → OpenCode subagent invocation via `@agent`
- `TodoWrite` → OpenCode todo/task tracking only if available; otherwise "maintain an explicit checklist"
- `CLAUDE.md / memory / docs` in reflect → `AGENTS.md / approved memory / docs`
- `~/.claude/projects/*.jsonl` transcript search in summary/summarizer → OpenCode-specific handoff summary based on visible session context and repo state

Keep references to Claude only in docs that describe the source plugin or dual-format maintenance.

## Open Questions

- Should generated OpenCode commands be unprefixed (`/make`) or namespaced (`/up-make`) if a user's project already has commands with the same names?
- Should `librarian` be read-only local/web only, or should the project also ship optional GitHub CLI guidance for authenticated private repo search?
- Should `summary` be kept in the first OpenCode release, or deferred until OpenCode session transcript access is defined?

Default answers for implementation:

- Use unprefixed commands.
- Make `librarian` read-only and tool-agnostic, with optional `gh`/web guidance in prose.
- Keep `summary`, but remove Claude transcript scraping and label it lower-confidence than Claude's current version.

## Verification

Run after implementing the generator:

```bash
python3 scripts/generate-opencode.py
find .opencode/agents .opencode/commands .opencode/skills .opencode/references -type f | sort
rg -n "up:|/up:|Task tool|subagent_type|~/.claude/projects|No skills directory|agent-only" .opencode/agents .opencode/commands .opencode/skills .opencode/references
```

Expected:

- `.opencode/agents/{explorer,implementer,researcher,reviewer,summarizer,librarian,finder,oracle,diff-explainer}.md`
- `.opencode/commands/{make,try,step-back,summary,reflect,explain-diff}.md`
- `.opencode/skills/<valid-name>/SKILL.md` for each process/discipline skill
- `.opencode/references/{brevity,principles}.md`
- No stale Claude-only invocation language in generated files
- Existing `plugins/up/` and `.claude-plugin/` unchanged

Manual OpenCode smoke:

1. Start OpenCode in the repo.
2. Confirm `/make`, `/reflect`, and `/explain-diff` appear.
3. Confirm `@finder`, `@librarian`, `@oracle`, and `@diff-explainer` are invokable.
4. Confirm generated skills appear in the skill tool description when enabled.

## Rollback

If OpenCode's markdown frontmatter rejects any generated agent permission structure:

- Keep generated markdown bodies.
- Move agent metadata into `opencode.json` generation.
- Leave `.opencode/agents/*.md` as prompt-only files referenced by JSON config.

If the skills layer proves too noisy:

- Keep `.opencode/skills/` as the source of reusable instructions.
- Tighten skill descriptions before considering flattening.
- Do not return to a single giant `make` prompt unless OpenCode skill discovery fails in practice.
